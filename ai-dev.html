10 Laws on AI-Powered Software Development

1. The Valley of AI Skepticism
You have to push through initial skepticism about AI coding tools. Most engineers resist because they think they can code faster themselves, but they're not measuring the actual time it takes to ship something complete.
2. Context Engineering Over Prompt Engineering
The real skill isn't crafting perfect prompts but providing the right context. Think of it as building up a relevant knowledge window for the AI rather than perfecting magical instructions. Most models are good enough now that even imperfect prompts work if the context is right.
3. The Three-Step Framework
Approach AI-assisted coding as Explore (gather context and investigate options), Plan (create detailed step-by-step instructions), and Build (execute the plan with validation). This shifts engineering from typing code to orchestrating its creation.
4. Test-Driven Development Renaissance
AI tools make TDD practical when most engineers avoided it before. Write tests first, then use them as context when asking the AI to implement the code. The AI's job becomes simply "make these tests pass" – a clear, measurable goal.
5. The Plan File Revolution
Create detailed Markdown plan files that outline exactly what needs to be built. Check these into your branch so reviewers understand your approach. This allows you to resume work later, switch models mid-task, and maintain clarity throughout the development process.
6. Monoliths Beat Microservices
AI tools work better with monolithic codebases than distributed microservices. When dependencies are spread across separate repos, the AI struggles to understand relationships between components. Consider mono-repos even if maintaining microservice architecture.
7. The 90% Threshold
When you get comfortable with these tools, 90% of your code can be AI-generated. This isn't about removing engineers but amplifying their capabilities – shifting from typing code to making technical decisions and understanding product trade-offs.
8. Commit Early and Often
When people complain AI tools "mess up and waste time," it's usually because they're not committing to Git frequently enough. If you keep going without saving state, you'll eventually hit a point where the AI makes a mistake and you lose all progress.
9. Staff Engineers Unleashed
These tools are most valuable for your most senior engineers, not juniors. Your staff engineers understand the system best but have limited time – AI tools let them implement exploratory features, load testing, and complex integrations they'd otherwise deprioritize.
10. Value Over Code Volume
The skill of an engineer is shifting from "how much code can you output" to "how much value can you produce." Technical decisions, product trade-offs, and communication skills become more valuable as code generation becomes accessible to everyone with the right training.

Summary: How I 10x My Engineering With AI

Core Philosophy

The key insight is rejecting "magic prompt" solutions in favor of matching AI workflows to different types of cognitive load. What used to take weeks of coding now happens in hours through strategic tool selection rather than universal solutions.

Productivity metrics achieved:

  • What took a full week now happens in hours
  • 100% of pull requests opened by AI tools
  • AI handles 30% of code reviews and debugs half of encountered bugs
Workflow 1: Flow-State Coding (Clear-Intent Implementation)

When to use:

  • You know exactly what "done" looks like
  • Tasks are small, clear, and well-scoped
  • Stakes are low and direction is clear
  • Working on single, focused tasks without breaking flow

Tools:

  • Windsurf/Cursor paired with "thinking models" (Claude Sonnet 4, Gemini 2.5 Pro)

Method:

  • Describe precise edits in plain English
  • AI implements across files while you maintain architectural control
  • You are the thinker, AI is the executor
  • Keep all design decisions to yourself

Key distinction from "vibe coding":

  • Works "inside-out": You know both destination and route, AI translates decisions into syntax
  • NOT "outside-in": Telling AI what you want and letting it figure out how
  • AI acts as your hands, not your brain

Benefits:

  • Speed without sacrificing quality or intent
  • Minimal context switching
  • Feels like pair programming with a tireless partner

Guardrails:

  • Keep diffs small and review everything
  • Don't outsource design decisions
  • Maintain complete architectural control
Workflow 2: Architecture/Discovery (Research and Reasoning)

When to use:

  • Blank canvas situations
  • Unclear "done" state
  • Complex interdependencies
  • Need to explore and discover approaches you wouldn't consider

Tools:

  • ChatGPT with o3 + tools
  • RepoPrompt or Claude Code agentic search (for full codebase context)
  • Multiple models in parallel (Claude 4 Opus, Gemini 2.5 Pro, o3)

Systematic methodology:

  1. Start broad with open questions ("What are my options under X constraints?")
  2. Run same prompts across multiple models
  3. Compare their different takes and framings
  4. Probe trade-offs and explore approaches
  5. Gradually narrow from exploration → plan → tasks
  6. Converge to a PRD/implementation plan

Benefits:

  • Surfaces approaches you wouldn't consider
  • Accelerates problem shaping
  • Each model provides different perspectives (ChatGPT: implementation difficulty, Claude: performance focus, Gemini: stability concerns)

Process notes:

  • Embrace AI as true thought partner
  • Start with ambiguity, avoid over-specification early
  • Goal is to be surprised by new solutions
  • Treat responses as rough drafts to synthesize
  • Validate with docs/tests before building
Workflow 3: Parallel Agents (CTO Method)

When to use:

  • Multiple independent tasks with crisp specs and boundaries
  • Architecture already decided
  • Components don't depend on each other to build

Tools:

  • Devin (creates full pull requests)
  • Claude Code with git worktrees
  • OpenAI's Codex (agent)
  • Cursor Background Agents

Implementation:

  • Break work into independent specifications
  • Spin up agents per task/branch using git worktrees for isolation
  • Rotate through reviews and feedback iteratively
  • Merge work streams progressively

Required skills:

  • Product Manager abilities: Breaking down features into clear, actionable specs with defined boundaries
  • Tech Lead skills: Reviewing code efficiently, spotting architectural issues, providing clear technical direction
  • Strong context-switching discipline

Benefits:

  • Compresses weeks into days via true parallelism
  • 5-10x productivity multiplier potential when mastered

Critical challenges:

  • Context switching is the main difficulty
  • Requires strong PM/tech lead discipline
  • Need to avoid hidden dependencies
  • Must enforce style/tests/CI gates
  • Tools and practices still evolving (experimental workflow)
Quick Decision Framework
  • Well-scoped, low risk task? → Flow-state coding (Cursor/Windsurf)
  • Fuzzy goal or architectural challenge? → Research mode (o3/Claude/Gemini + repo search) → produce PRD
  • Multiple independent, spec'd tasks? → Parallel agents with separate branches and tight review loops
Essential Practices

Quality control:

  • Keep diffs small and reviewable
  • Insist on tests and benchmark checks in AI-generated PRs
  • Define "definition of done" for all agent work
  • Maintain input/output contracts and guardrails
  • Enforce lint, unit/integration tests, and CI gates

Research optimization:

  • Run same research prompt across 2-3 models to expose blind spots
  • Compare different framings and synthesize comprehensive approaches
  • Start wide, then gradually narrow to actionable plans

Project management:

  • Maintain evolving scratchpad/PRD: exploration → plan → tasks
  • For parallel work, ensure clear boundaries and no hidden dependencies
  • Rotate review cycles efficiently to maintain momentum
Meta-Principle

The goal isn't to automate thinking but to clear space for deeper thinking. By matching workflow to cognitive load type, you can leverage engineering experience while offloading mechanical development tasks. This creates space to focus on what requires actual thought and creativity.

The approach is anti-hype: no magic prompts or one-size-fits-all solutions, just strategic tool selection based on the specific type of cognitive work required.

Summary: Here’s how I use LLMs to help me write code

1. The Over-Confident Intern Model
Treat LLMs as lightning-fast, over-confident pair programming assistants who excel at tedious tasks and looking things up, but will make inhuman mistakes. Don't anthropomorphize their failures—they're tools, not humans.
2. Context Engineering Trumps All
Master context management over prompt crafting. Success comes from feeding the right existing code, examples, and conversation history rather than perfecting magical instructions. When conversations stop working, wipe the slate clean.
3. The Research-Then-Dictate Pattern
Start exploratory with "what are my options?" to leverage the LLM's breadth, then switch to authoritarian mode with exact specifications like function signatures. Be the architect, let the LLM be the typist.
4. Training Cutoffs Shape Tool Selection
Consider model training dates when choosing libraries. Stick with stable, well-documented tools that were popular before the cutoff date, or be prepared to feed the LLM recent examples as context.
5. Test Everything, Trust Nothing
You cannot outsource testing to the machine. If you haven't seen it run, it's not working code. The human responsibility is delivering working systems—manual QA is non-negotiable.
6. Conversations Beat One-Shots
Expect multiple iterations. A bad initial result isn't failure—it's a starting point. Use follow-ups like "write that better" or "break out that repetitive code" to refine toward what you actually want.
7. Sandboxed Execution Changes Everything
Choose tools that can safely run and iterate on code (ChatGPT Code Interpreter, Claude Artifacts). The run-code-in-a-loop pattern is so powerful it should drive your tool selection.
8. Speed Unlocks Ambition
The biggest win isn't faster completion of planned work—it's building projects you wouldn't have justified spending time on at all. LLMs make throwaway experiments and learning accelerators economically viable.
